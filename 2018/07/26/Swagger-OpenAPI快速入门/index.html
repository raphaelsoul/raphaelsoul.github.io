简介

Swagger是一款开源的api文档书写工具， 要求编写者按照一定的语法规范来描述一个API的所有接口，包括请求方法、请求路径、请求头、 cookie、等等一切内容

目前常见的版本为2.0, 同时， 升级到3.0后， 语法结构发生了比较大的改变，并且重命名为openapi。 本文以openapi的标准来介绍swagger的用法。因为目前SwaggerHub、SwaggerEditor
以及Swagger Specification使用的都是openapi 3.0.1的规范。

工具

IDE/编辑器集成

我们可以使用多种工具来编写swagger文档，一般书写都是.yaml格式， 我们可以转换成.json格式来是自动化方案和文档生成工具能够更方便地处理其中的数据。

  1. 官方提供的在线的SwaggerEditor

    注意 SwaggerEditor的语法检查规范是openapi v3的规范， 如果需要编写swagger v2规范的接口描述，可以使用SwaggerEditor2

  2. Swagger Plugin for IntelliJ
    可以直接在JetBrains全系列的IDE的插件管理中直接搜索Swagger Plugin安装， 主要提供了swagger/openapi的语法自动补全。

  3. openapi-lint for VSCode
    直接在VSCode中安装。提供了代码自动补全和lint功能，语法错误的地方会在控制台的Problem标签页中警告

自动化集成

同时我们可以通过各种语言/平台的开源工具集成，通过自动化测试和自动化部署，测试接口，并部署静态文档。具体参考这里选择你需要的集成类库。

结构简介

一个最简单的结构

  1. info 基本信息

    即整个文档的meta信息 都在info节中。除了必要的version、 title之外 还可以有description、 termsOfService等字段

  2. servers 服务端点

    用来描述服务器端点(Endpoint), 可以有多个

  1. paths 路由描述

    即所有的路由 是我们主要要编写的部分。 以下示例包含了大多数常用的描述

  2. components 可重用组件

    组件 主要是用来通过$ref来重用的描述. 使用方式$ref: '#/components/{type}/{Name}'

    3.1 schemas 可重用的数据模型
    3.2 parameters 可重用的参数描述
    3.3 securitySchemes 可重用的鉴权方式
    3.4 requestBodies 可重用的请求body
    3.5 responses 可重用的响应
    3.6 headers 可重用的头部声明
    3.7 examples 可重用的示例
    3.8 links 可重用的链接
    3.9 callbacks 可重用的回调
